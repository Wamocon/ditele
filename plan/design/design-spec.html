Design System v4

The system was already specified and it was good. It never reached the browser. Two bugs made 761 utility classes and both webfonts silently do nothing. This page is built from the same tokens the app uses, so everything you see here is what the app looks like once they are fixed.

01 — Root causes

Two bugs, both invisible

Neither produced a warning, an error, or a failed build. Tailwind accepted the input and emitted CSS the browser then discarded.

Bug 1 — Tailwind v4 variable syntax761 sites · 104 files

Tailwind v4 changed the arbitrary-CSS-variable shorthand from square to round brackets. Compiled through this project's own @tailwindcss/postcss@4.3.3:

/* text-[--color-fg-muted] — the v3 form still in the codebase */
.text-\[--color-fg-muted\] { color: --color-fg-muted; }      /* invalid → discarded */

/* text-(--color-fg-muted) — the v4 form */
.text-\(--color-fg-muted\) { color: var(--color-fg-muted); }  /* correct */

Bug 2 — circular font variablesglobals.css · 2 lines

@theme emits --font-heading: var(--font-heading) into :root. next/font sets the real value on a class on the same element. Specificity ties, :root comes later, so it wins — and a self-referential custom property is invalid at computed-value time. 

.rosario_…_variable { --font-heading: "Rosario", "Rosario Fallback"; }  /* byte 4717  */
:root, :host        { --font-heading: var(--font-heading); }        /* byte 15897 ← wins */

Headings fall through to Georgia, serif; body falls through to system-ui. Rosario and Raleway have never rendered. Both are humanist sans faces, so the serif fallback is also a category error. 

/* the fix — inline resolves at use sites, no :root declaration is emitted */
@theme inline {
  --font-heading: var(--font-heading);
  --font-body:    var(--font-body);
}
The rule

-(--token) for every design token. Never -[--token]. Never a raw hex outside globals.css. WS-D0 lands an ESLint rule that fails the build on the old form.

04 — Glass & depth

Glass chrome, flat data

The header above this text is the recipe, live. Scroll and watch content pass under it. Depth comes from the relationship between floating glass and crisp opaque content — not from shadows on everything. 

.glass {
  background: var(--glass-bg);
  backdrop-filter: blur(var(--glass-blur)) saturate(var(--glass-saturate));
  -webkit-backdrop-filter: blur(var(--glass-blur)) saturate(var(--glass-saturate));
  border: 1px solid var(--glass-border);
  box-shadow: var(--glass-shadow), inset 0 1px 0 0 var(--glass-highlight);
}
/* Translucency is decoration. Legibility is not. */
@supports not (backdrop-filter: blur(1px)) { .glass { background: var(--color-bg); } }

05 — Motion

Two tiers, one vocabulary

A technique allowed in one tier is a bug in the other. Motion that charms on visit one is friction on visit fifty, and admins are on visit fifty.

Showcase tier

(public) · (auth) — landing, catalog, about, faq, login

Expressive. Scroll choreography, parallax, tilt, gradient text, magnetic CTA. Up to 520ms.

App tier

(student) · (trainer) · (admin)

Calm and fast. Nothing over 320ms. Nothing that moves under the pointer.

Scroll-driven: native first, JS fallback

Native scroll timelines run off the main thread and cannot jank. The bar at the very top of this page uses one.

@supports (animation-timeline: view()) {
  .reveal-native {
    animation: fade-in-up var(--duration-slow) var(--ease-out) both;
    animation-timeline: view();
    animation-range: entry 0% entry 60%;
  }
}

No animation libraries. No Framer Motion, no GSAP, no Lenis. CSS plus small React primitives. The dependency list stays frozen.

07 — Account menu

The missing control

Today app-header.tsx:99 renders the avatar as a <span> with a title attribute. No menu, no profile link, and no way to sign out from admin or trainer at all — signOutAction exists only on the student profile page.

Click the avatar in the header to see the target. It uses the native Popover API, so top-layer rendering and light-dismiss come for free.
