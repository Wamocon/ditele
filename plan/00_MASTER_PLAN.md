## 1. Executive summary & honest verdict

### What we are building

**DiTeLe V3** ÔÇö a digital technical learning platform for WAMOCON Academy. Students learn software-testing theory (video, PDF, quiz) and practice (scenario Ôćĺ external test website in an iFrame Ôćĺ defect report Ôćĺ trainer review). Trainers review submissions and answer questions. Admins author content and run the platform.

### The decision in one line

> **Keep the Supabase database exactly as it is. Delete the entire frontend. Rebuild it with the WAMOCON brand, a top-nav shell, and a reusable component library.**

### Scope: the whole application

We build **all of it** ÔÇö every route, every role, every feature in the P0 and P1 sets, properly, to a real-launch standard. No time-boxing, no "good enough for a demo". The work is organised into **phases and gates**, not hours. A phase is done when its gate passes, however long that takes.

| | |
|---|---|
| **What we build** | 42 routes across Guest / Student / Trainer / Admin ┬Ě DE + EN + RU ┬Ě full learn Ôćĺ submit Ôćĺ review loop ┬Ě content authoring ┬Ě user, cohort and enrolment management ┬Ě Q&A ┬Ě notifications ┬Ě ratings ┬Ě bug reports ┬Ě certificates |
| **Quality standard** | Real launch. Every route has loading, empty and error states. Works at 375px and 1440px, light and dark, keyboard-navigable. Every data boundary validated. Nothing hardcoded. |
| **Deliberately deferred** | Gamification, AI coach, Labs, Portfolio, Integrations, multi-tenancy, SSO, analytics dashboards ÔÇö each blocked by an external decision nobody has made yet (┬ž15). Not a time cut, a correctness cut. |

### The four preconditions

Everything else in this plan is detail. These four are not:

1. **OneDrive is paused, or the repo is moved out of it.** Three requirement docs vanished from the working tree during the analysis for this plan and had to be restored from git. That was one process reading files. Many chats writing thousands of files will be worse.
2. **Wave 0a completes fully before any parallel chat starts.** If chats start against an unverified foundation you get several incompatible half-apps. Wave 0a is serial, one chat, and it has a hard gate.
3. **The RPC signatures are verified, not guessed.**
4. **The database contains real course content to render.**

Miss any one and the work does not converge.

### Infrastructure runs in parallel, not after

`NEXT_PUBLIC_SUPABASE_URL` currently points at `http://192.168.178.75:56721` ÔÇö a private LAN address, plain HTTP, non-standard port. Real students cannot reach it and secure auth cookies will not persist. This is infrastructure, not code, and no amount of React fixes it.

It is completely independent of the build: different person, different skills, zero shared files. **Start it alongside Wave 0.** [01_LAUNCH_READINESS.md](01_LAUNCH_READINESS.md) has the full checklist.

### What "good" looks like, not just "done"

A plan that treats 42 routes as equal produces 42 equally mediocre screens. Budget quality deliberately:

| Tier | Screens | Standard |
|---|---|---|
| **Signature** | Landing ┬Ě **Task workspace** ┬Ě Trainer review detail | Every animation, every empty state, every micro-interaction. Should feel expensive. Spend polish time here without guilt. |
| **Core** | Student dashboard ┬Ě Course detail ┬Ě Review queue ┬Ě Catalog ┬Ě Admin dashboard | Clean, consistent, well-spaced. No custom flourishes needed. |
| **Functional** | All admin CRUD ┬Ě settings ┬Ě profile ┬Ě history ┬Ě notifications | Correct and fast, components used as-is. **Plain is fine.** Do not spend time here. |

### The north star

Theory delivery ÔÇö video, PDF, quiz ÔÇö is a commodity. Every LMS on earth does it and nobody would pick DiTeLe for its video player. **The practice loop is the entire product:** scenario Ôćĺ live test website in an iFrame Ôćĺ structured defect report Ôćĺ trainer review.

The question that decides whether this platform is worth building is not "did we ship all 16 requirements." It is: **is the practice loop noticeably better than doing the same exercise with a spreadsheet and a second browser tab?**

Three consequences for the build:
1. `IframePanel` gets real engineering, not a bare `<iframe>` ÔÇö resize, fullscreen, reload, open-in-tab, honest mobile fallback (┬ž8.3).
2. The defect form is designed like a professional bug tracker (severity, steps, expected, actual). The students are learning to write real defect reports ÔÇö **the form is part of the teaching.**
3. The trainer review screen shows the submission next to the task, with hints used and time taken, so a review takes 90 seconds instead of five minutes. Trainer throughput is what makes the platform economically viable.

## 4. Supabase ÔÇö server-hosted backend

### 4.1 It is on the server, not local

The Supabase instance runs on the **WAMOCON server at `192.168.178.75`**, not on your laptop and not in Supabase Cloud. It is reached over the local network.

```
NEXT_PUBLIC_SUPABASE_URL = http://192.168.178.75:56721
```

**Verified reachable on 2026-07-21:**
- `GET /auth/v1/health` Ôćĺ `200 OK`
- `GET /rest/v1/` Ôćĺ PostgREST OpenAPI, PostgreSQL 14.10
- Anon role sees 2 tables (`courses`, `course_localizations`) + 2 RPCs ÔÇö correct, RLS is doing its job
- Service role sees **99 tables + 48 RPCs** ÔÇö the full schema is deployed

### 4.2 Environment variables ÔÇö READ THESE, DO NOT INVENT THEM

All credentials already exist in **`.env.local`** at the repo root. It is git-ignored. **Never print its values, never commit it, never hardcode a key in source.**

| Variable | Purpose | Exposure |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase endpoint on the server | Browser-safe |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Anonymous key, RLS-protected | Browser-safe |
| `SUPABASE_SERVICE_ROLE_KEY` | Privileged key | **SERVER ONLY.** Never import into a Client Component. |
| `DITELE_AUTH_RATE_LIMIT_HMAC_KEY` | HMAC for login throttling | Server only |
| `DITELE_APP_ORIGIN` | Canonical origin for CSRF + auth redirects | Server only |
| `DITELE_DATA_MODE` | `supabase` ÔÇö keep it | Server only |
| `DITELE_AI_PROVIDER` / `_LAB_` / `_INTEGRATION_` | `disabled` ÔÇö keep them disabled | Server only |

> ÔÜá´ŞĆ **DO NOT run `npm run env:local`.** That script targets a local loopback Supabase and will overwrite your server configuration. The file header says so explicitly.
> ÔÜá´ŞĆ **DO NOT run `npm run db:reset`, `supabase start`, or `supabase db push`.** The remote database is the live one. Resetting it destroys the schema every chat depends on.

### 6.6 Motion

Nine keyframes. That is the whole animation vocabulary.

```css
@keyframes fade-in      { from { opacity: 0 } to { opacity: 1 } }
@keyframes fade-in-up   { from { opacity: 0; transform: translateY(8px) }  to { opacity: 1; transform: none } }
@keyframes fade-in-down { from { opacity: 0; transform: translateY(-8px) } to { opacity: 1; transform: none } }
@keyframes scale-in     { from { opacity: 0; transform: scale(.96) }       to { opacity: 1; transform: none } }
@keyframes slide-in-right { from { transform: translateX(100%) } to { transform: none } }
@keyframes slide-up     { from { transform: translateY(100%) } to { transform: none } }
@keyframes shimmer      { from { background-position: -200% 0 } to { background-position: 200% 0 } }
@keyframes pulse-ring   { 0% { box-shadow: 0 0 0 0 rgb(187 39 52 / .38) } 70%,100% { box-shadow: 0 0 0 12px rgb(187 39 52 / 0) } }
@keyframes progress-fill{ from { width: 0 } }
```

| Interaction | Animation |
|---|---|
| Page main content mount | `fade-in-up` 320ms `--ease-out` |
| List/grid items | `fade-in-up`, stagger `index * 40ms`, **capped at 240ms** |
| Card hover (desktop only) | `translateY(-2px)` + `--shadow-md`, 200ms |
| Button press | `scale(0.97)`, 120ms |
| Modal / dialog | `scale-in` 200ms; backdrop `fade-in` |
| Mobile sheet | `slide-up` 320ms `--ease-out` |
| Toast | `slide-in-right` desktop, `slide-up` mobile |
| Skeleton | `shimmer` 1.6s infinite |
| Progress bar | `progress-fill` 600ms on mount |
| Nav active indicator | 200ms width/position transition |
| Unread notification dot | `pulse-ring` 2s infinite |

**Mandatory, in `globals.css`:**

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

Never animate `width`, `height`, `top` or `left` in a loop. Only `transform`, `opacity`, `box-shadow`.

### 7.3 Scrollbar ÔÇö thin overlay Ôťů DECIDED

**Decision: thin styled overlay scrollbar.** No chunky OS scrollbar, but the affordance stays. Users can still see how long a page is and still drag it.

  border-radius: 999px;
  transition: background var(--duration-base) var(--ease-out);
}
*::-webkit-scrollbar-thumb:hover { background: var(--color-brand); }

.scroll-area { overflow-y: auto; overscroll-behavior: contain; }
```

Two supporting affordances stay, because they are good regardless:
1. A **2px red scroll-progress bar** pinned under the header, width = `scrollY / scrollHeight`.
2. Internal scrolling panes get a **fade mask** at the bottom edge while more content exists below.

This keeps the clean look, stays WCAG-friendly, and costs nothing.

### 9.3 Permission enforcement ÔÇö three layers, all mandatory

| Layer | Where | What it does |
|---|---|---|
| **1. Database (authoritative)** | Supabase RLS + `SECURITY DEFINER` RPCs | The real boundary. Already built and deployed. |
| **2. Server** | `requireRole()` in every role layout + every Server Action | Redirects/403s before rendering. Never trust the client. |
| **3. Client** | Nav visibility, disabled buttons | **Cosmetic only.** Hiding a button is not security. |

```ts
// src/shared/auth/guard.ts ÔÇö WS-0 owns this
export async function requireRole(allowed: UiRole[]): Promise<Principal> {
  const principal = await getPrincipal();          // throws if no session
  if (!allowed.includes(toUiRole(principal.roles))) redirect("/403");
  return principal;
}
```

Called in `app/(student)/layout.tsx`, `app/(trainer)/layout.tsx`, `app/(admin)/layout.tsx`. Every Server Action re-checks ÔÇö a layout guard does not protect a POST.

## 13. Data layer contract

### 13.1 The rule

> **No page calls Supabase directly.** Every read and write goes through a typed function in `src/shared/data/`.

This is what makes six parallel chats possible. Each workstream owns its own data file; nobody duplicates a query; a change to a shape breaks the build loudly instead of failing silently at runtime.

```
src/shared/data/
  rpc.ts          # ÔşÉ WS-0: typed wrappers for every P0 RPC + verified signatures
  session.ts      # WS-0: getPrincipal, requireRole, signIn, signOut, register
  catalog.ts      # WS-1
  learning.ts     # WS-2
  questions.ts    # WS-3
  profile.ts      # WS-3
  notifications.ts# WS-3
  review.ts       # WS-4
  content.ts      # WS-5
  admin.ts        # WS-6
```

### 13.2 Shape every function follows

```ts
// Never throw into a page. Always return a discriminated result.
export type Result<T> =
  | { ok: true; data: T }
  | { ok: false; error: { code: string; message: string; retryable: boolean } };

export async function getMyLearningCourse(courseId: string): Promise<Result<LearningCourse>> {
  const supabase = await createServerClient();
  const { data, error } = await supabase.rpc("get_my_learning_course", { p_course_id: courseId });
  if (error) return { ok: false, error: mapPostgrestError(error) };
  return { ok: true, data: LearningCourseSchema.parse(data) };
}
```

- Every RPC return is validated with **zod** at the boundary. An unexpected shape fails visibly, in one place, with a useful message.
- `mapPostgrestError` (WS-0) turns Postgres error codes into German user-facing messages: `42501` Ôćĺ "Keine Berechtigung", `23505` Ôćĺ "Bereits vorhanden", `PGRST116` Ôćĺ "Nicht gefunden".
- Server Actions return `Result<T>` too. Pages render `ErrorState` on `ok: false`.

### 13.3 ÔÜá´ŞĆ WS-0's most important job: verify the RPC signatures

The argument names above (`p_course_id`) are an **assumption**. WS-0 must introspect the live database and record the real signatures:

```sql
select p.proname, pg_get_function_arguments(p.oid) as args,
       pg_get_function_result(p.oid) as returns
from pg_proc p join pg_namespace n on n.oid = p.pronamespace
where n.nspname = 'public' order by p.proname;
```

The result goes into `plan/status/RPC_CONTRACTS.md`. **If six chats each guess argument names, six chats each fail.** This single step is worth more than any other 20 minutes in the build.

### 13.4 Rendering rules

- **Server Components by default.** Fetch on the server, in parallel (`Promise.all`), never waterfall.
- **Client Components only for:** forms, dialogs, the iframe panel, timers, autosave, the theme toggle, the mobile sheet. Mark them `"use client"` and keep them leaf-level.
- **Mutations are Server Actions**, never client `fetch` to an API route. `revalidatePath` after a write.
- **URL holds the state** for filters, tabs, pagination and selected ids.
- **`loading.tsx` + `error.tsx` in every route segment.** A route without them is not done.
- Timestamps stored UTC, rendered with `Intl.DateTimeFormat("de-DE")`.
