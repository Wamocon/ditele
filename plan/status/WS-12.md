## The decision that shapes this workstream

**Every number on the board comes from one `security definer` RPC, and that RPC
serves the trainer view too.**

WS-11's handoff is the reason for the first half: `xp_ledger`, `badge_awards`
and `learner_streaks` are **self-read only** under RLS, so an admin session
reads **zero rows** from all three. `RPC_CONTRACTS.md` §10 makes it worse than
that — an **admin reads 0 rows from `attempts` and 0 from `submissions`**, and a
**trainer reads 0 from `enrollments`** (I-018). There is no session on this
deployment that can read what the board needs, so the definer function is not a
convenience; it is the only construction that works.

A board built from direct reads would have rendered every learner at Level 1,
no streak, no tasks done — and **it would have looked like data**. That is the
`[]`-not-an-error trap `RPC_CONTRACTS.md` §10 calls the most expensive bug in
this codebase, and this screen is the worst place in the app for it: a progress
board wrongly reporting "no activity for 7 days" is indistinguishable from one
working correctly and reporting a stalled learner. It is the reason
`ws12-browser-check.mjs` asserts a **non-zero** XP value rather than asserting
that the page rendered.

The second half — one RPC for both roles — is `06_…` §8 item 4. Two functions
that are supposed to agree eventually do not. One function that scopes by the
caller cannot disagree with itself.

## Things I learned that are written down nowhere else

1. **A progress board must never refresh a streak, and it is a one-line
   mistake.** `app_private.refresh_learner_streak` looks like a read — it is
   named like one, and the Arena hub calls it on page load. It **awards XP**:
   daily-activity and milestone payments, inside the refresh. Calling it per row
   from an admin dashboard would pay real XP to every learner on the board
   because an administrator opened a page, which is exactly the "gamification
   pointing the wrong way" failure `05_…` §6 names. The board reads the stored
   row instead, so the streak is as fresh as the learner's own last Arena visit.
   **That is a deliberate accuracy trade, not an oversight.**

2. **`security definer` inverts where authorization lives, and the probe has to
   invert with it.** Every other read in this codebase is protected by RLS, so
   "can a learner see this?" is answered by a policy. `list_progress_board`
   bypasses RLS by construction — the scope clause inside the function is the
   *only* thing stopping a learner reading the whole cohort's progress. So
   `ws12-board-probe.sql` §4 asserts a learner reads **zero** rows, and the flag
   probe §3/§4 assert a learner and a trainer are both refused. Those are not
   defensive extras; without them nothing in the system says a leak would be
   noticed.

3. **The `[]`-not-an-error trap has a worse form than the one documented.**
   `RPC_CONTRACTS.md` §10 warns that a silently-empty read "reads as no data
   rather than forbidden". On a progress board it is worse than ambiguous — it
   is *plausible*. "No activity, Level 1, no streak" is a completely reasonable
   thing for a real dashboard to say about a real learner, so the broken version
   and the working version are visually identical. The only defence is asserting
   a **non-zero** value against a learner known to have ledger rows, which is
   what the browser check does.

4. **Two of three browser failures were the check being wrong, and both came
   from the same root cause: `DataTable` renders every row twice.** There is a
   `hidden md:block` table and a card list below it, so a bare locator finds 12
   controls for 6 rows and `.first()` picks whichever copy is `display:none` at
   the current width. `boundingBox()` then returns null, which reads as "the
   control is missing" rather than "you measured the invisible copy". Use
   `:visible` in every assertion against this component. The other failure was
   measuring a 44px touch target at 1280px, where this codebase *deliberately*
   relaxes to a 36px desktop rhythm — the shared Button's `size="sm"` documents
   the same policy. **Measure touch targets at 375px or you will report the
   design system as a bug.**

5. **A page-wide `waitForSelector('[role=status]')` races against the app
   shell.** There is already an empty aria-live region in the layout, so the
   wait resolves instantly against *that* and reads `""` before the Server
   Action has returned. The check failed while the feature worked. Scope the
   wait to the container the action renders into — here `details[open]`, where
   `ActionMessage` renders nothing while idle, so the element's existence is
   itself the completion signal.

6. **"Behind" needed a threshold, and `<` was the wrong one.** `05_…` §G10
   writes *"completed ≪ elapsed schedule"* — two less-than signs, and the
   difference is the whole feature. A learner one task behind their own schedule
   is a normal Tuesday; flagging them trains an administrator to ignore the
   column, at which point the badge is worse than absent. The shortfall
   threshold is 2, named as a constant in the SQL rather than buried in a
   predicate, so the next person can argue with it.

7. **The one course with no trainer is the case that mattered.** Six of seven
   active enrollments sit on a single course with a single trainer; the seventh
   has none. I would have shipped a form that silently notified nobody on that
   row if the browser check had not clicked the top-risk learner — who happened
   to be exactly that one. **The highest-risk row is the row the board sorts
   first, so it is also the row a manual check clicks first. That is luck, and
   it should not have been.** Seed the awkward case deliberately (I-058).
