## The decision that shapes this workstream

**Every number on the board comes from one `security definer` RPC, and that RPC
serves the trainer view too.**

WS-11's handoff (`plan/status/WS-11.md`, item 4 of its RESUME block) is the
reason for the first half: `xp_ledger`, `badge_awards` and `learner_streaks` are
**self-read only** under RLS, so an admin session reads **zero rows** from all
three. A progress board built from direct table reads would silently show every
learner at Level 1 with no streak — the `[]`-not-an-error failure mode
`RPC_CONTRACTS.md` §10 calls the most expensive bug in this codebase. It is
worse here than elsewhere, because a progress board that reads "no activity" is
indistinguishable from a progress board that is working and reporting a stalled
learner.

`RPC_CONTRACTS.md` §10 makes it broader than the reward tables: an **admin
session reads 0 rows from `attempts` and 0 from `submissions`**, and a
**trainer reads 0 from `enrollments`** (I-018). There is no session on this
deployment that can read everything the board needs. So the definer function is
not a convenience — it is the only construction that works.

The second half — the same RPC serving both roles — is `06_…` §8 item 4:
"migrate the trainer view to read `enrollments` directly, or admin and trainer
will show different numbers and nobody will trust either." One function that
scopes by the caller's own role is a stronger guarantee than two functions that
are supposed to agree.

## Ownership note — the trainer page is NOT mine

`06_…` §8 item 4 asks WS-12 to unify the trainer view. `06_…` §4 does **not**
assign me `src/app/[locale]/(trainer)/trainer/progress/**` or
`src/shared/data/review.ts`, and §4 is listed under "Rules that override
everything else". So the split is:

- **I build** the unified source of truth — one RPC, role-scoped, plus
  `listProgressBoard()` in `src/shared/data/progress.ts` — and prove it returns
  correct rows *for a trainer session*, not only an admin one.
- **I do not edit** the trainer page. Swapping its import is one file in
  someone else's tree; it goes to `ISSUES.md` for WS-13.

That keeps the boundary and still delivers the thing item 4 actually wants,
which is that the two screens cannot disagree.
