### ⭐ `supabase gen types` needs Docker even with `--db-url`

WS-8, WS-11 and WS-12 each concluded the generated types could not be
regenerated because `npm run db:types` passes `--local` and starts a Docker
container. The obvious next thought — point it at the remote database instead —
**does not work either**, and WS-13 measured it rather than assuming:

```
$ npx supabase gen types --lang typescript --db-url "postgresql://…:56722/postgres?sslmode=disable"
Connecting to 192.168.178.75 56722
failed to connect to the docker API at npipe:////./pipe/dockerDesktopLinuxEngine
```

`gen types` runs `pg_meta` in a container regardless of where the database
lives. So hand-editing is the only option on this machine, and I-052/I-057 are
closed by hand-adding the six missing objects, introspected from the live
database with psql. A real regeneration overwrites them with the same content.

### Applying a migration — the shortest path that works

WS-11's is the good one; psql 18 is installed locally and the port answers:

```bash
PGPASSWORD=postgres "/c/Program Files/PostgreSQL/18/bin/psql.exe" \
  "postgresql://postgres@192.168.178.75:56722/postgres?sslmode=disable" \
  -v ON_ERROR_STOP=1 -f supabase/migrations/<file>.sql
```

## Things I learned that are written down nowhere else

1. **`supabase gen types --db-url` still needs Docker.** Three workstreams
   stopped at "`db:types` is `--local`, and there is no Docker". The natural fix
   is not a fix: `gen types` starts a `pg_meta` container whatever database you
   point it at. Worth knowing before anyone spends time on it a fourth time.
2. **A `null`-returning Server Component is safe to render but not safe to
   tab.** `HuntPanel` was deliberately built to return `null` for non-hunts so
   it could be dropped in unconditionally — and that is exactly right for a
   stacked layout. In a tab bar the same property produces a tab with nothing
   behind it. "Safe to render unconditionally" and "safe to *offer*
   unconditionally" are different claims.

### ⛔ NOT DONE — cohort must disappear from the interface

**This is the next task, and it is bigger than it looks.** Stopped here
deliberately rather than started and abandoned half-way.

The coordinator's requirement: *"there is no group, no cohort, just only course
and tasks."* Three screens still contradict it:

1. **`/admin/applications`** — the card's field is labelled "Course" (WS-13
   relabelled it) but its VALUE is `row.cohortName`, so it prints "Release 0
   Cohort" where a course name belongs. The course is already the card's
   subtitle. **Relabelling was not enough and made it worse**: the label now
   promises a course and shows a cohort.
2. **`/trainer/submissions`** — a "Cohort" filter dropdown and a "Cohort"
   column, both real and both wrong under the new vocabulary.
3. The assign dialog picks a cohort, because `assign_enrollment` requires one.

**Why this is not a find-and-replace.** `cohorts` is a live table. It carries
`content_version_id` — which is how a learner's snapshot is resolved — and
`assign_enrollment` is the only path that creates a `cohort_memberships` row.
Removing the concept from the DATA layer would break enrolment entirely. The
correct shape is almost certainly:

- keep `cohorts` as an invisible implementation detail, one per course;
- drop the cohort column and filter from both screens;
- make the assign action take a COURSE and resolve the cohort itself, choosing
  the single active cohort for that course and failing loudly if there is more
  than one.

⚠️ **Check the multi-cohort case before writing any of it.** If a course ever
has two active cohorts, "assign to the course" is ambiguous and the resolution
rule has to be a product decision, not an implementation guess. Today:
`select course_id, count(*) from public.cohorts where state = 'active' group by 1`.
