## Reconnaissance — facts established before writing any code

These were measured against the live database, not assumed.

1. **`bug_categories` holds exactly ONE row** — `functional`, global
   (`organization_id is null`), `labels = {de,en,ru}`. The design names five
   (functional · UI · data · performance · a11y). The other four do not exist.
2. **A learner reads ZERO rows from `bug_categories`.** Its only policy is
   `bug_categories_content_write`, gated on `has_permission('content.manage')`.
   There is no read policy. So a label picker that queries the table renders
   empty for every student — the failure mode ISSUES I-003/§10 warns about,
   where RLS returns `[]` and it reads as "no data" rather than "forbidden".
3. **`hunt_scenarios` has ZERO rows.** WS-8's slice pointed a hunt task at
   `external_id='checkout-v1'` but never seeded a scenario. So there is no
   ground truth on the database yet; WS-9 seeds it. WS-10's matching must
   degrade honestly when the scenario is missing, not blow up.
4. **The storage bucket `task-evidence-private` exists** and is private.
   `create_task_evidence_upload_intent` is granted to `authenticated`;
   `finalize_task_evidence_upload_service` is granted to `service_role` **only**
   — and that grant is real, contrary to the blanket claim in
   `RPC_CONTRACTS.md` §0.5 that service_role holds no RPC grants. §0.5 is about
   *table* privileges. Screenshots are therefore feasible.
5. **`task-workspace.tsx` is not mine, and does not have to be.** It renders
   `DefectForm`, calls `formatDefectReport` and stores the result through
   `attempt_drafts.evidence_draft`, which is `jsonb`. Adding fields to
   `DefectReportSchema` and to the form flows through the existing draft and
   submit machinery **without touching one file WS-10 does not own** — provided
   every new prop on `DefectForm` is optional, because the existing call site
   will not pass it.

### ⭐ A privilege leak this workstream nearly shipped, and how it was caught

`decide_hunt_finding` was first written to authorize with
`app_private.can_access_submission` — the helper WS-8's own RLS policy on
`hunt_findings` uses, and the obvious choice.

**It is the wrong helper, and probing as each role is what proved it.** That
function governs *read* access on the review screen, and a learner may of course
read their own submission. So the learner who filed a bug report could call
`decide_hunt_finding` on it and mark it `confirmed` — the verdict WS-11's award
engine is specified to pay XP for. A student could grade their own work.

The fix is to make the check *identical* to `public.decide_submission`'s, read
out of the deployed function rather than guessed:

```
actor_is_trainer := app_private.is_active_cohort_review_trainer(actor, cohort, org);
actor_can_manage := app_private.has_permission('cohort.manage', org, cohort);
if not actor_is_trainer and not actor_can_manage then raise 42501; end if;
```

**Grading is not a read.** Anyone adding a further arena write RPC should copy
`decide_submission`'s pair, not `hunt_findings`' RLS predicate.

### Regression evidence — the non-hunt submission path is untouched

Run against **real rows on the live database**, inside a transaction that was
then rolled back (verified: `hunt_findings` is back to 0 rows):

| Probe | Expected | Result |
|---|---|---|
| Hunt submits | 1 pending finding, summary from the draft or the answer text | ✅ 1 row |
| **Practical submits** | **findings unchanged** | ✅ **unchanged** |
| Hunt re-submits | upsert, not a second row; wording refreshed | ✅ 1 row, `row_version` 2 |
| `confirmed` with no planted code | refused | ✅ `22023` |
| Trainer confirms with a code | succeeds, `decided_by` set | ✅ `row_version` 2 |
| Replay of the same verdict | no version bump | ✅ unchanged |
| Stale `p_expected_version` | errors **immediately** | ✅ `40001`, **no hang** |
| Learner decides own finding | refused | ✅ `42501 review scope denied` |
| Trainer decides | succeeds | ✅ |

The stale-version row matters on its own: `RELEASE.md` §6 item 4 (I-007/I-009)
records that a stale expected-version **hangs** on this deployment, Kong 504s
and the PostgREST pool is unusable for ~30 s afterwards. `decide_hunt_finding`
is a plain guarded `UPDATE … where row_version = p_expected_version` with no
lock and no retry loop, and it returns `40001` in milliseconds. **Whatever
causes that hang, this function does not reproduce it.**

### ⭐ Nobody can read the filed report — not the trainer, and not the student

`06_…` §8 asks for a ticket view "as the student and trainer both see it" and a
panel showing "whether all required fields are present". Both need the
**structured** report, not the prose blob in `submissions.answer_text`.

The structured report lives in `attempt_drafts.evidence_draft`. The deployed
policy — there is exactly one on that table — is:

```
attempt_drafts_owner_read  FOR SELECT USING (
  attempt.learner_id = auth.uid()
  AND attempt.state IN ('in_progress', 'revision_required') )
```

So:

* **A trainer can never read it.** Not in any attempt state. The ground-truth
  panel would have had nothing structured to match against.
* **The student loses it the moment they submit.** The state predicate excludes
  `submitted`, `resubmitted` and `accepted`. A learner cannot re-read their own
  filed ticket — a ticket view built on that table shows an empty page exactly
  when the report matters most.

Neither is discoverable from the design docs; both were found by reading the
policy. The rejected alternative was parsing the report back out of
`answer_text` — that blob is assembled from **translated** field labels, so a
parser keyed on German strings would silently start returning nothing the day
the en/ru pass lands, which `RELEASE.md` §8 lists as the next session's first
task.

Fixed by carrying a faithful copy on the finding row, written by the trigger
that already reads the draft inside the submit transaction. Verified end to
end: a draft holding all nine ticket fields produced a finding whose
`reported_details` carried the labels array, the environment string and the
steps intact — then rolled back.

## Deferred / not built

### ⛔ Unit 7 — screenshot attachment. NOT BUILT.

`06_…` §8 item 2 asks for screenshot upload reusing `evidence_uploads`. The
**data half is done and shipping**: `DefectReport.screenshotIds` exists, is
`.default([])`ed, round-trips through `evidence_draft`, is copied onto
`hunt_findings.reported_details`, is counted in `formatDefectReport`, and the
ticket view renders a count badge for it. **What does not exist is the upload
itself.** Nothing writes to `screenshotIds`, so the badge never appears.

Reconnaissance done, so the next chat does not repeat it:

* The bucket `task-evidence-private` **exists** and is private.
* `create_task_evidence_upload_intent(p_attempt_id, p_title,
  p_original_file_name, p_declared_mime_type, p_declared_byte_size,
  p_client_sha256, p_idempotency_key, p_correlation_id)` is granted to
  `authenticated` and returns `(upload_id, bucket_id, object_key,
  upload_state, rejection_code, expires_at, replayed, correlation_id)`.
* `finalize_task_evidence_upload_service(...)` is granted to **`service_role`
  only** — and that grant is real. `RPC_CONTRACTS.md` §0.5 says service_role
  holds no RPC grants; **§0.5 is about table privileges and overstates the
  case.** Verified by reading the migration's own `grant execute` line.

**The one hard blocker, and it is a real one.** The upload is a three-step
dance — intent → PUT the bytes to Storage → finalize — and the finalize step
must run as `service_role`. That means a server action or route handler holding
the service key, doing its own MIME/byte verification. That is a **new
privileged code path in an app whose evidence flow already passed a security
review**, and `05_…` §G3 is explicit that WS-10 must not invent a second upload
path. Building it in the context that remained would have meant shipping a
privileged handler without the care it deserves.

**Recommendation:** do it as its own focused unit, not as the tail of another.
Everything downstream of it already works, so it is purely additive — no
migration, no schema change, no change to anything committed here.

## Things I learned that are written down nowhere else

1. **`can_access_submission` is a READ helper and must never gate a write.**
   A learner passes it on their own submission. Using it to authorize
   `decide_hunt_finding` would have let a student mark their own bug report
   `confirmed` and collect the XP WS-11 pays for it. Copy
   `decide_submission`'s pair — `is_active_cohort_review_trainer` **or**
   `cohort.manage` — for any new arena write. Probing as each role is what
   found this; reading the code did not.
2. **Nobody can read a filed defect report after submit.**
   `attempt_drafts_owner_read` requires `learner_id = auth.uid()` AND state in
   `('in_progress','revision_required')`. A trainer never qualifies; the
   student stops qualifying the instant they submit. Anything that needs the
   structured report after submit must carry its own copy —
   `hunt_findings.reported_details` is WS-10's.
3. **`submissions` is UNIQUE on `attempt_id`.** One submission row per attempt;
   a resubmission UPDATEs it. Any "on submit" hook must sit on
   `submission_versions`, which gets one row per actual submit event, or it
   fires once and never again.
4. **A scenario's authoring prose broke flat token matching.** WS-9's
   `expected` field is a long German paragraph — for the decoy, four lines
   about why the behaviour is *not* a bug. Under one flat token bag a short
   student report could not cover thirty tokens, so **richer authoring made
   matching worse**. Score each description *separately* and take the best.
5. **`innerText` returns RENDERED text, so `uppercase` CSS changes it.** An
   early run of `ws10-browser-check.mjs` reported seven failures against a
   panel that was working perfectly. Fold case in any assertion against this
   app's section labels.
6. **A `.replace()` that "worked" may have matched only part of what you
   intended.** Wiring the panel for testing, the import landed and the JSX did
   not — the file is CRLF and the pattern was written with `\n`. The guard was
   `assert s2 != s`, which the import alone satisfied. **Assert the occurrence
   count, not just that something changed.** The browser check caught it.
7. **WS-8's "kill by port, not by name" is correct and cost time anyway.**
   `pkill -f "next start"` kills the shell wrapper; the node child keeps port
   3110 and silently serves the OLD build, so a fresh check tests stale code.
   Use `Get-NetTCPConnection -LocalPort 3110 | Stop-Process`.
8. **`RPC_CONTRACTS.md` §0.5 overstates its case.** "The service-role key
   cannot read or write tables" is about *table* privileges;
   `finalize_task_evidence_upload_service` **is** granted to `service_role`.
   Check the grant before concluding a service RPC is unusable.
