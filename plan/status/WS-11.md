### Applying a migration — a shorter path than the one WS-8 documented

WS-8 routes everything through `ssh Nvidia-1 'docker exec … psql'`. That is not
necessary from this machine:

```bash
PGPASSWORD=postgres "C:/Program Files/PostgreSQL/18/bin/psql.exe" \
  "postgresql://postgres@192.168.178.75:56722/postgres?sslmode=disable" \
  -v ON_ERROR_STOP=1 -f supabase/migrations/<file>.sql
```

`psql` 18 is installed locally and the port answers over the LAN.
`?sslmode=disable` is mandatory (I-039) — without it the CLI fails in a way that
looks exactly like a wrong password. Running the file through `-f` rather than
`-c "…"` also sidesteps the quoting mangle WS-8 hit, and leaves CRLF handling
inside psql instead of in a shell pipeline.

## Things I learned that are written down nowhere else

1. **`xp_ledger` could not express the reward system it was built for.**
   `skill_id` was `not null` with an FK to `skills`, and the deployment holds
   exactly one skill — so a streak bonus, which has no skill, was unstorable.
   `source_kind` was a CHECK over four values, none covering a streak or a daily
   activity. Both design documents describe `xp_ledger` as "idempotency already
   built in — we write the RPC, not the model". The idempotency is indeed there
   and it is excellent. The **shape** needed two changes before anything could
   be written at all. Check the columns, not just the constraints.

2. **`postgres` holds `rolbypassrls`, and every reward table has FORCE ROW LEVEL
   SECURITY.** That combination is why the shipped
   `decide_submission_effects_unowned` can insert into `notifications` even
   though `notifications` has one policy and it is not an INSERT policy. The
   consequence for anyone adding an `app_private` writer: the
   `alter function … owner to postgres` line is **load-bearing, not
   boilerplate**. A `security definer` function owned by any other role is
   silently blocked by its own tables — FORCE RLS means even the owner is
   subject to the policies.

3. **Badge idempotency needs a DERIVED source event, not a random one.**
   `badge_awards` is unique on `(badge_id, learner_id, source_event_id)`. A
   threshold badge is re-evaluated on every accepted review, so a fresh uuid per
   evaluation awards "Erster Fund" again on every subsequent acceptance —
   forever. `md5('badge:'||badge_id||':'||learner_id)::uuid` turns that unique
   index into exactly-once, which is what a threshold badge actually means. The
   same trick gives streak milestones and daily activity their stable keys.
   `05_…` §G5's sketch implies a varying source event; for threshold rules that
   is wrong.

4. **A function prop cannot cross into a Client Component, and the failure is
   invisible until runtime.** `CelebrationBanner` took
   `levelTitle: (level) => string`. React refused the entire render with
   "Functions cannot be passed directly to Client Components", which in a
   production build reaches the browser as a **digest-only error with no
   message** — the page just says "Etwas ist schiefgelaufen". `typecheck`,
   `lint` and `next build` are all green either way. The Server Components in
   `hub-sections.tsx` take formatter props quite happily, so the pattern looks
   established right up until it is used one file over. Pass templates, not
   formatters, across the boundary.

5. **A bridged streak day must not be counted, and the probe is what forced the
   decision.** §8.4 defines the streak as "die Anzahl aufeinanderfolgender Tage
   **mit mindestens einer Lernaktivität**" — days with activity. A freeze keeps
   the chain intact; it does not credit a day nobody studied. So 0,1,2 · gap ·
   4,5 is **5**, not 6. My probe asserted 6 first, failed, and that is how the
   choice got made deliberately rather than by whichever line I happened to
   write. Between a count that is honest and one that flatters, take honest.

6. **`attempts_active_task_uidx` permits exactly ONE live attempt per
   `(enrollment_id, task_id)`** — partial, over
   `in_progress|submitted|revision_required|resubmitted`. Anything fabricating
   attempt history has to use `abandoned` (or vary the task). Worth knowing
   before writing a seed.

7. **`hunt_findings_bonus_has_no_code`** (WS-8's) forbids a `bonus` verdict from
   naming a planted code — correct, because "bonus" *means* a bug we did not
   plant. Setting `verdict='bonus'` without clearing `planted_code` fails the
   check, and the award engine never sees the row.

8. **`const URL = …` at module scope silently breaks
   `new URL(…, import.meta.url)`** in a plain `.mjs` script — "URL is not a
   constructor", which reads like a Node version problem. Every other
   `scripts/ws*.mjs` gets away with it only because none of them resolves a file
   path. Name the Supabase URL something else.

9. **The XP price list exists twice and has to.** `public.xp_rules` is what
   pays; `XP_RULES` in `model.ts` is what the hub renders, and it must be
   client-reachable because the level meter is a Client Component while
   `data.ts` is `server-only`. Two copies of a price list drift, and the drift
   is quiet: the hub says 20 XP, the ledger pays something else, and
   `rule_version` — whose entire job is that an old award keeps its provenance —
   starts describing points it never recorded. `scripts/ws11-check-rules.mjs`
   is what makes the duplication safe. It reads as a **learner**, so a missing
   read policy fails loudly instead of comparing against an empty `[]`.

10. **Running the browser check against the live database awards real rewards.**
    The first green run gave `learner@ditele.local` 5 XP for `daily_activity`
    and the `first-approval` badge — off a submission that was accepted back in
    V3. That is the feature working, not test pollution, but it means the hub's
    numbers move when you check it. The two SQL probes roll back; the browser
    check does not, and cannot, because it exercises the real path.
