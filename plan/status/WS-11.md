## ⚠️ Read this before touching the database

### Applying a migration — a path that works from THIS machine, no ssh

WS-8's note routes everything through `ssh Nvidia-1 'docker exec … psql'`. That
is not the only way, and it is not the shortest:

```bash
# psql is installed locally; the database port is reachable over the LAN.
PGPASSWORD=postgres "C:/Program Files/PostgreSQL/18/bin/psql.exe" \
  "postgresql://postgres@192.168.178.75:56722/postgres?sslmode=disable" \
  -v ON_ERROR_STOP=1 -f supabase/migrations/<file>.sql
```

`?sslmode=disable` is mandatory (I-039). Running the file through `-f` rather
than `-c "…"` also sidesteps the quoting mangle WS-8 hit, and it keeps CRLF
handling inside psql instead of in a shell pipeline.

### The three shipped-schema facts that shape this workstream

Measured, not assumed — all three change what the award engine can be:

1. **`xp_ledger.skill_id` is `not null` with an FK to `public.skills`**, and the
   database holds exactly **one** skill. Neither design document mentions this.
   A streak bonus or a daily-activity award has no skill to attribute, so the
   column has to become nullable (0 rows today, so the alter is free) or the
   engine has to invent taxonomy. See unit 2.
2. **`xp_ledger.source_kind` is a CHECK over four values** —
   `accepted_submission`, `validated_evidence`, `mastery_gain`,
   `completed_mission`. Streaks fit none of them; the CHECK gets widened.
3. **`postgres` holds `rolbypassrls`, and every reward table has FORCE RLS on.**
   So a `security definer` function owned by `postgres` writes freely, which is
   exactly how the shipped `decide_submission_effects_unowned` already inserts
   into `notifications` and `mastery_events`. A definer function owned by
   anything else would be silently blocked by its own tables.
